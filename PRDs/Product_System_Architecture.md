# 法律 AI 助手：系统架构与核心逻辑开发指南

## 1. 项目愿景

开发面向律师行业的提效工具，优先解决三类痛点：

1. **数值与规则计算易错**：违约金、利息、滞纳金等依赖法条与合同约定的分段计算。
2. **多媒体证据处理繁琐**：录音录像转写、聊天截图排序与事实归纳耗时长。
3. **行政与时效节点易漏**：查封续封、开庭、举证期限等需登记与提前提醒。

本文档描述 **MVP 范围**、**模块边界**、**确定性计算与 LLM 的职责划分**，供后续接口与实现对照。与端侧分期、搜索、知识库展示口径以《[UI_Design_Spec.md](./UI_Design_Spec.md)》**互为引用、同步更新**。民间借贷与房屋租赁的**法条/公式级口径**以《[Legal_Logic_Implementation.md](./Legal_Logic_Implementation.md)》为准；三份 PRD 与 `docs/HANDOFF.md` 中涉及同一能力时**以 PRD 为产品口径源**，HANDOFF 侧重续作命令与路径索引。

## 2. 产品范围与分期建议

### 2.0 第一期与第二期边界（当前结论）

**知识库 / 向量检索 / RAG**

- **架构上保留**扩展位：例如在后端预留「文档块 + 向量检索」的数据模型或配置开关、异步任务管道，避免第二期推倒重来。
- **第一期不落地、不启用**：不部署向量库、不跑 Embedding 流水线、不接依赖检索增强的问答产品形态；开发与联调精力优先放在 **框架与页面**、**金额计算与提醒**相关链路；案件列表、证据/文书等主链路 **按迭代纳入**（与《[UI_Design_Spec.md](./UI_Design_Spec.md)》一致）。

**金额类法律规则计算**

- **业务流程与字段口径尚未与业务定稿**，需与律师/业务方讨论后再冻结（例如：入口是「纯手工填参」还是「合同抽取 + 人工确认」、是否多笔本金并行、出函格式等）。
- **第一期允许**：先把 **页面壳子、列表/详情、计算任务占位数据结构** 做好；计算结果在业务未定稿前可用 **占位数据或简单表单** 演示交互。**当前 workbench 原型例外**：已在 `legal_calc` 中落地 **民间借贷、房屋租赁** 的确定性计算与导出（规则版本见 `legal_calc/version.py`），并与 Web 原型联调；全案由、合同抽取、案件表等仍按迭代纳入（与下表及《[Legal_Logic_Implementation.md](./Legal_Logic_Implementation.md)》一致）。

**首发形态**

- 当前以 **单律师账号测试** 为主，复杂多租户权限可后置。

| 模块 | 第一期侧重 | 第二期及以后 |

业务与规则冻结后，可将下列能力视为「完整 MVP」目标，与 2.1 第一期范围区分使用。

| 模块 | MVP 能力 | 刻意不做（可更晚阶段） |
|------|-----------|-------------------|
| 法律规则计算引擎 | 单案多笔本金、固定规则/LPR 分段、输出可审计明细 JSON | 全量案由模板、自动判例引用 |

### 2.3 成功标准（可验收）

**第一期（框架期）**

- 主链路页面可跑通：**以当期迭代为准**。若以 **金额计算与提醒** 为优先，则至少包含计算相关页与提醒/待办相关页的导航与空态，并符合《[UI_Design_Spec.md](./UI_Design_Spec.md)》。**案件列表/详情、证据或文书入口** 可按迭代纳入，不强制与计算同周交付。
- **全局/列表搜索**：前后端均 **待定**，第一期不强制实现（见 UI 规范 §2.1、§2.5）。
- 与业务未定稿的模块：**不强制**数值与公式验收，以「结构可接、无伪权威展示」为准。

**规则与计算定稿后（与 2.2 全量 MVP 对齐）**

- 任意一笔利息类计算：**数字仅由脚本产出**，可追溯每一步公式与区间。
- 任一案关键日期：**有来源字段**（OCR 文本或人工录入）与**推算规则版本**。
- 证据时间轴：**每条节点可链接到原始文件与时间来源**（文件内时间戳或人工标注）。

## 3. 核心功能模块

### 3.A 法律规则计算引擎（Deterministic Engine）

**当前状态（与业务对齐前）**：计算**端到端业务流程**（从哪一步录入、是否经过抽取、如何出函/导出）**待与业务方讨论定稿**。本节描述的是**定稿后**应坚持的技术原则，实现可分期上线。

**目标**：在律师确认输入的前提下，对违约金、利息、滞纳金等做**可重复、可审计**的计算。**当前仓库**已覆盖 **民间借贷利息** 与 **房屋租赁滞纳金 + 占用费**（见 `legal_calc/private_lending.py`、`legal_calc/rental/`），其它案由仍按产品迭代扩展。

**职责划分**：

1. **结构化提取（LLM 辅助）**  
   从合同或案情描述中抽取候选字段：本金、利率约定类型、起止日、还款节点等。输出 **JSON Schema 校验通过** 的结构，**不**把最终金额当作模型输出事实。

2. **确定性计算（禁止 LLM 算数）**  
   使用 Python（或经测试的 JS）实现法定/约定公式，例如 LPR 浮动、分段计息、逾期天数等。单元测试覆盖边界日（闰年、月末、利率变更日）。

3. **输出**  
   分段明细（起止日、适用利率、本金余额、本段利息、累计）、**规则版本号**（便于法条或内部规则更新后复算）。

**实现要点**：

- 采用「提取 → 校验 → 计算」三阶段 API；计算阶段 **只读** 已通过校验的数值字段。
- 规则与法条以配置或代码常量版本化（如 `lpr_policy_v202405`），避免散落魔法数。

### 3.B 辅助行政助手（Admin Automation）

**目标**：减轻法院文书信息录入负担，并对续封等关键节点做提醒。

**关键路径**：

1. **OCR**：对查封通知书、传票等扫描件/PDF 做版式与文字识别；输出全文 + 版面块坐标（便于人工框选纠错）。
2. **日期与规则推算**：在人工确认「查封日期」「标的类型」等后，按内部维护的规则表计算「建议续封截止日」（例如区分动产/不动产等——**具体条线与期限须由你们法务定稿后写入规则表**）。
3. **提醒**：到期前多档提醒（如 30 天、15 天）；通道可选微信订阅消息、短信或站内信，需用户授权与模板审核。

**实现要点**：

- OCR 与业务库解耦：先存 `ocr_job` 结果，再进入 `case_event` 人工确认流。
- 所有推算结果存 `computed_at`、`rule_set_id`，支持事后复核。

### 3.C 证据链处理中心（Evidence Pipeline）

**目标**：将杂乱素材整理为可引用的「事实时间线」草稿。

**能力**：

1. **多媒体转写**：对接 ASR（录音/视频音轨），输出带时间码的转写稿。
2. **证据时间轴**：对图片类证据可 OCR 提取界面时间戳（置信度标注）；LLM 仅做「要点摘要」类生成，**不改变原始文件**。

**实现要点**：

- 原始文件对象存储（S3 兼容或云厂商 OSS），数据库仅存元数据与哈希。
- 时间轴节点类型：`transcript_segment` | `ocr_timestamp` | `manual`，便于过滤与导出。

## 4. 逻辑架构（概念）

```
[ Web / 小程序 ]
        |
        v
[ API 网关 / 鉴权 ]
        |
        +-- [ 案件与任务 CRUD ] ------> PostgreSQL
        |
        +-- [ 计算服务 ] --> 纯函数库 + 规则配置（无 LLM）
        |
        +-- [ 异步任务队列 ] --> OCR / ASR / 大模型抽取
        |         |
        |         v
        +-- [ 向量检索 ] ----> 第二期：Chroma 等，知识库与 RAG（第一期不部署，仅占位）
```

**原则**：凡是涉及**金额、日期法律效力**的结论，以**规则引擎 + 人工确认**为准；LLM 输出一律带 `confidence` 或「需确认」标记。第一期未上知识库时，**不得**用大模型闲聊冒充「所内权威依据」。

## 5. 技术栈建议

| 层级 | 建议 | 说明 |
|------|------|------|
| 后端 | Python（FastAPI） | 便于科学计算、测试、与 OCR/ASR SDK 集成 |
| 前端 Web | React + Ant Design Pro 或 Ant Design + 自建布局 | 与《[UI_Design_Spec.md](./UI_Design_Spec.md)》对齐；**原型阶段**可为 Vite + Ant Design 单页 + Tabs（见 §2.4）；分期与搜索口径以两文档同步更新为准 |
| 移动端 | 微信小程序或 Uni-app | 拍照上传、订阅消息与金刚区入口 |
| 关系库 | PostgreSQL | 案件、文书事件、计算快照、审计字段 |
| 向量库 | **第二期再引入**（如 ChromaDB、PGVector） | 第一期架构预留即可，**不部署、不联调 RAG** |
| 对象存储 | 云 OSS / MinIO | 证据与 OCR 中间件 |
| 任务队列 | Celery + Redis 或 RQ | 异步 OCR、ASR、长文本抽取 |

## 6. 数据域（MVP 最小实体）

以下名称供建模参考，实现时可英文化表名。

- **案件（Case）**：案号、当事人脱敏字段、状态、负责人。
- **计算任务（CalculationRun）**：输入 JSON、规则版本、输出明细 JSON、创建人、不可变快照。
- **文书与 OCR（Document / OcrResult）**：文件指针、全文、版面、识别引擎版本。
- **时限事件（CaseEvent）**：事件类型、来源、关键日期、推算结果、提醒计划。
- **证据项（EvidenceItem）**：类型、存储 URL、哈希、时长（音视频）。
- **时间轴节点（TimelineNode）**：关联证据、时间来源、摘要文本、是否已人工确认。

## 7. 安全与合规（MVP 必做底线）

- **传输与存储**：HTTPS；敏感字段加密或脱敏展示；对象存储私有桶 + 临时签名 URL。
- **权限**：当前首发以 **单律师账号测试** 为主（与 UI 规范登录极简一致）；后续扩展单所多用户或 SaaS 时再加强；数据模型可预留 `org_id`。
- **审计**：谁、何时、对哪条计算/事件做了确认或修改，写入审计表。
- **日志**：禁止在日志中打印完整身份证号、银行卡号等；LLM 调用侧做 PII 过滤策略（可二期加强）。

## 9. Cursor 开发首步指令（示例）

**业务未定稿前（优先框架）**

> 请根据架构文档第一期范围，优先搭建 **金额计算** 与 **提醒/待办** 相关占位页；使用假数据或本地 Mock API；数据模型预留 `input_json`、`rule_version`、`output_json` 字段，计算按钮可先返回静态示例 JSON。**案件列表/详情** 若当期不做可延后；**不要**实现无需求支撑的顶栏/列表搜索。便于与业务讨论展示形态。

**计算规则与流程已定稿后**

> 请根据本项目架构，编写一个 Python 纯函数模块：接收「起始日期」「金额」「利率规则标识与参数」，实现分段累加计息，返回包含每一分段起止日、适用利率、本段利息与合计的 JSON；并附 pytest 覆盖闰年、利率变更日、金额为 0 的边界用例。**不要**在计算路径中调用大模型。

## 10. 待与产品/法务确认的事项

1. **金额与利息等计算全流程**：入口字段、是否经合同抽取、复核节点、导出格式——**待与业务讨论后写入流程图与字段说明**。
2. **续封等期限规则**：银行类 1 年、房产 3 年等表述是否覆盖全部案由？是否有地域或程序例外？（可与计算模块分期定稿。）
3. **租户模型**：当前以单律师测试为主；后续若扩展单所多用户或 SaaS 多所，再升级鉴权与数据隔离。
4. **微信能力**：是否确定只做小程序，还是需要服务号/H5 共用一套账号体系？
5. **知识库与 RAG**：已定为**第二期**；第一期仅保留架构扩展点，不投入向量库与入库流水线。
6. **全局/列表搜索**：是否纳入第一期，与《[UI_Design_Spec.md](./UI_Design_Spec.md)》§2.1、§2.5 统筹决定；未定时前后端均不强制交付。

确认以上事项后，可将「规则表」与「权限模型」从本文档拆成独立《规则配置说明》与《权限矩阵》附录。
